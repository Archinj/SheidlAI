Process

From threat to fix in four steps

1

Scan

ShieldAI runs continuous monitoring across 12 attack vectors: login attempts, file changes, plugin CVEs, traffic anomalies, security headers, database permissions, and more. No configuration needed — it starts watching the moment you activate it.

2

Analyze

When a risk is detected, the AI agent interprets the raw data and generates a specific, actionable fix — not a generic warning. A brute force attack becomes a precise firewall rule. A CVE becomes a targeted update recommendation with severity context and patch availability.

3

Approve

Every proposed action appears in the approval queue with: what triggered it, the exact change it would make (full diff), the risk of acting vs. not acting, and a confidence score. You approve, edit, schedule, or dismiss. No action is ever taken without your explicit decision.

4

Harden

Approved changes are deployed instantly via WordPress and server APIs. Every deployed action is logged with a timestamp and one-click rollback available. Your site's security posture improves continuously — without you needing to understand the technical details.

Under the Hood

How ShieldAI is built

ShieldAI is a WordPress plugin that wires together four mature technologies — no research-level AI required. Here's exactly how each layer works.

WordPress Plugin Shell
Standard WordPress plugin

ShieldAI is distributed as a standard WordPress plugin (.zip) installable from the WP admin. It uses WP's native hook system, settings API, and custom post types — no non-standard server access required on activation.

PHP 8.1+ WP 6.0+ OOP / PSR-4
Data Collection Layer
No external agents needed

All data is already inside WordPress. Login attempts live in wp_ tables. File checksums are compared against WordPress.org's published hashes. Installed plugins are queried via the WP Plugins API. No server-level access required.

wpdb WP Filesystem API get_plugins()
Agent Scheduler
WP-Cron + server cron fallback

Each agent is registered as a scheduled WP-Cron event with its own cadence. For reliability on low-traffic sites, a server-side cron ping (configurable) triggers WordPress cron externally — eliminating WP-Cron's dependency on site visits.

wp_schedule_event() Action Scheduler
LLM Integration
Claude API / OpenAI GPT-4o

Agents send structured JSON payloads (threat data, file diffs, CVE details) to the LLM API with precision-engineered prompts. The LLM returns structured JSON — a specific action, plain-English explanation, confidence score, and severity level. No free-form text enters the execution pipeline.

Claude API JSON mode wp_remote_post()
Approval Queue
Custom DB table + WP admin UI

Every agent recommendation is written to a custom shieldai_queue table with status, payload, diff, trigger context, and severity. The WP admin dashboard renders this as a review queue. No action is written to WordPress until a row's status is set to approved.

$wpdb->insert() WP Settings API REST API
Executor + CVE Feeds
WP APIs + public vulnerability data

Approved actions are dispatched to typed executor classes: HtaccessExecutor, PluginUpdateExecutor, HeaderExecutor. Vulnerability data is sourced from WPScan's free API and NVD's public CVE feed — queried weekly against the site's plugin inventory.

WPScan API NVD CVE feed .htaccess

End-to-end data flow

What stays on your server

  • ✓ All WordPress data, logs, and file system contents
  • ✓ The approval queue and all pending recommendations
  • ✓ Executed firewall rules and configuration changes
  • ✓ Full audit log of every agent action

What leaves your server (anonymized)

  • → Structured threat summary sent to LLM API for analysis
  • → Plugin slugs + versions sent to WPScan CVE API
  • → No raw logs, no customer data, no PII ever leaves
